1. Cloud Architecture and Scalability

This pipeline is engineered using Microsoft Azure's PaaS and Serverless offerings to provide elastic scalability while mathematically driving idle costs to zero.

Ingestion Layer: Azure Functions

The production tier (EP1) keeps minimum instances eternally warm, eliminating cold-start delays and enabling complex VNet integration with IP firewalls while dynamically auto-scaling to thousands of containers during the 8:00 AM boot-storm window when 1M+ endpoints power on simultaneously.

Why ADLS Gen2 over Standard Blob Storage? The Delta Lake framework relies on atomic rename capabilities at the filesystem level. Standard Blob Storage emulates directories using a flat namespace, which cannot support Delta transaction logs (_delta_log) reliably. ADLS Gen2 provides true POSIX directory architecture required for transactional integrity.

2. Telemetry and Enterprise Logging

  • Azure Functions: The function_app.py script routes standard Python logging directly to Azure Application Insights. All 202 Accepted and 500 Server Error results are natively indexed for SLA monitoring.
  • Azure Databricks: The scd2_processor.py utilizes structured Python logging with timestamps. Databricks inherently hooks into the stdout pipeline, capturing these entries directly into the Driver Logs for real-time observability.

3. Deployment Security Posture

  1. System-Assigned Managed Identity: The Azure Function App relies on a Microsoft Entra ID Managed Identity. No connection strings, SAS tokens, or static passwords are hardcoded anywhere in the codebase.
  2. Role-Based Access Control (RBAC): The Bicep template dynamically assigns Storage Blob Data Contributor to the Function App's Service Principal ID.
  3. Private Subnets: The Databricks workspace is injected into a dedicated Virtual Network with public and private delegation subnets. Public IP is disabled.
  4. Key Vault Integration: All secrets (storage account keys, function keys) are stored in Azure Key Vault and accessed via RBAC — never embedded in source code.

4. PySpark Data Processing Engine (ETL)

The core scd2_processor.py notebook executes specialized algorithms to parse, flatten, and historize raw JSON telemetry:

4.1 Two-Tier Deduplication

Intra-Batch Filtering: A PySpark Window function identifies and selects only the single most recent payload for a given endpoint within that execution cycle.

windowSpec = Window.partitionBy("EndpointId").orderBy(col("IngestionTimestamp").desc())
silver_df = clean_df.withColumn("rank", rank().over(windowSpec)) \
    .filter(col("rank") == 1).drop("rank")

Cross-Batch Hashing: PySpark computes a deterministic SHA-256 hash of the fully parsed payload. During the Delta Merge, it silently discards incoming rows if the data identically matches the active Gold table record.

exclude_cols = {"EndpointId", "IngestionTimestamp", "MessageId", "QuarantineReason", "IsActive", "ValidFrom", "ValidTo"}
hash_cols = [c for c in sorted(silver_df.columns) if c not in exclude_cols]
silver_df = silver_df.withColumn("PayloadHash", sha2(to_json(struct(*[col(c) for c in hash_cols])), 256))

4.2 First-Normal-Form Array Flattening

Endpoints transmit software and driver lists as heavily nested JSON arrays. The processor dynamically detects and atomizes these arrays into cleanly flattened rows suitable for Power BI dimensional modeling.

for field in silver_df.schema.fields:
    if isinstance(field.dataType, ArrayType):
        silver_df = silver_df.withColumn(field.name, explode_outer(col(field.name)))

4.3 Graceful Error Interception

Standard Spark behavior throws hard crash exceptions when reading empty directories or encountering rotating access keys (403). The processor wraps the engine in custom exception handlers, allowing the cluster to gracefully skip missing data without terminating the job.

try:
    raw_df = spark.read.json(raw_path)
except AnalysisException as e:
    if "Path does not exist" in str(e):
        logger.info("Landing zone is empty (no recent files). Skipping.")
        return
