Enterprise Endpoint Inventory Pipeline

Deployment Process Guide
Author: Ragotham Kanchi Version: 2.0 Date: March 2026 Classification: Internal

1. Introduction

This document outlines the operational process flow for capturing hardware, software, driver, and custom inventory data from 1M+ client endpoints and ingesting it into a centralized Azure Databricks Delta Lake with full SCD Type 2 dimensional history.

Figure 1 — End-to-End Production Architecture with VNet Boundaries and Unity Catalog Governance

2. Process Flow Architecture

Phase 1: Edge Collection

A PowerShell agent (inventory_agent.ps1) executes on the client endpoint via Intune, SCCM, or Group Policy on a daily schedule.

1
Payload Assembly: The agent queries WMI and system registries to build 4 distinct JSON payloads: Hardware, Software, Drivers, and Custom.
2
SHA-256 Hash Gate: Before transmitting data to the cloud, the agent computes a SHA-256 hash of each JSON payload and compares it to the locally stored hash from the previous execution. If the data is identical, transmission is skipped entirely — saving significant bandwidth across 1M+ endpoints.
3
Jittered Transmission: If the hash differs, the script applies a randomized startup delay (0 to 300 seconds) to prevent API stampedes, then fires an HTTP POST to the Azure Function endpoint carrying only the modified payload.

Phase 2: Ingestion Landing Zone

The Azure Function (function_app.py) receives the HTTP POST request and routes it to the data lake.

Why Azure Functions Elastic Premium (EP1)? Endpoint inventory traffic is highly uneven. When 1M+ laptops power on at 8:00 AM, the EP1 tier instantly scales to thousands of containers to absorb the POST requests, then releases them by 8:15 AM. Pre-warmed instances eliminate cold-start latency while VNet integration enables secure network isolation.
1
Authentication: Validates the Function Key via the x-functions-key header.
2
Extraction: Extracts EndpointId and PayloadType from the JSON body.
3
Enrichment: Injects server-side IngestionTimestamp (UTC) and MessageId (UUID v4) for deduplication and lineage tracking.
4
Storage: Streams the raw JSON securely into ADLS Gen2 using the Function App's System-Assigned Managed Identity.
Why ADLS Gen2 over Standard Blob Storage? Standard Blob Storage uses a flat namespace and emulates directories. Delta Lake relies on atomic rename and filesystem-level locking for its _delta_log transaction history. ADLS Gen2 provides a true POSIX Hierarchical Namespace, which is required for transactional integrity.

Landing Zone Path Structure:

raw/{PayloadType}/YYYY/MM/DD/{EndpointId}_{UUID}.json

Phase 3: Medallion Processing Engine

An Azure Databricks Job cluster activates on a daily schedule and executes the scd2_processor.py notebook.

Why Job Clusters? Interactive clusters are designed for 24/7 data-science exploration and cost significantly more per compute hour. Job clusters boot up, execute the ETL pipeline, and terminate immediately — minimizing billable uptime. The production tier autoscales Standard_D8ds_v4 worker nodes (2 to 8) based on payload volume with Unity Catalog governance enabled.

Bronze Layer — Raw Validation and Error Interception

  1. Reads all new JSON files from the landing zone using spark.read.json.
  2. Custom AnalysisException handlers gracefully skip empty directories and 403 Access Denied errors without terminating the cluster.
  3. Identifies payloads containing PySpark _corrupt_record fields (malformed JSON) and routes them to a dedicated quarantine Delta table.
  4. Records missing EndpointId are also quarantined with reason tagging.

Silver Layer — Two-Tier Deduplication and Flattening

  1. Intra-Batch Deduplication: In mobile environments, endpoints may send duplicate payloads due to network retries. PySpark Window functions partitioned by EndpointId select only the most recent payload per execution cycle.
  2. Array Normalization: Dynamically flattens nested JSON arrays (e.g., InstalledSoftware) into top-level columns via explode_outer for First Normal Form (1NF) compatibility.
  3. Deterministic Hashing: Generates a SHA-256 hash of the entire payload for cross-batch deduplication during the Gold merge phase.

Gold Layer — SCD Type 2 Delta Merge

  1. Cross-references incoming records against active records in the Gold Delta tables (dim_hardware, dim_software, dim_drivers, dim_custom).
  2. Performs a cryptographic comparison: source.PayloadHash != target.PayloadHash. If data is identical, the record is silently skipped.
  3. If an endpoint's data has changed, the existing active record is expired (IsActive = False, ValidTo = current_timestamp()).
  4. The new payload is inserted as a fresh record with IsActive = True and ValidFrom = current_timestamp().
  5. Schema auto-evolution is enabled via spark.databricks.delta.schema.autoMerge to accommodate new JSON fields without manual DDL.

3. Disaster Recovery and Fallback

If the Databricks cluster is unresponsive or unavailable, the Edge Agent continues to deposit JSON payloads into the ADLS Gen2 raw zone. No data is lost. Upon the next successful cluster startup, Databricks processes the accumulated backlog of JSON files. The idempotent SCD2 merge logic ensures no duplicate dimension rows are created even if the same data is processed multiple times.

Important: The quarantine Delta table should be monitored via Azure Monitor alerts. Sustained quarantine growth may indicate a schema change at the edge agent level or network corruption during transmission.
